# issue reviewer — 着手前 issue の判定 spec(8 観点 / 3 ファミリー)

**対称性の正確な言明(実装者の混乱を防ぐ・issue #93)**: 本ファイルは **dispatch ラッパ層では PR reviewer と対称** — `roles/issue-reviewer-dispatch.md`(薄い dispatch wrapper)+ 本ファイル(判定 spec)の 2 ファイル構成は、PR 側の `roles/pr-reviewer-dispatch.md` + `roles/pr-reviewer.md` と同型。**ただし内部分解層では非対称** — 本ファイル 1 つが、PR 側の `collectors/angles/*`(検出 rubric・9 ファイル)と `roles/pr-reviewer.md`(判定 / 整形 / 投稿)の**両役割を担う**。issue reviewer は単一エージェント判定のため、PR 側の候補収集(orchestrator が pr reviewer dispatch 前に走らせる `collectors/strategy.md`)に相当する前処理は無い。**実装者への含意**: collectors 風の観点分解(観点ごとの別ファイル)を issue 側へ無理に持ち込まない。観点のプラガブル化(PR 側 collectors の `.harness/collectors/angles/` 追加機構・#65 相当)は将来 follow-up で、本 spec では 8 観点を inline で固定する。

## これは何で、何をしないか

整った構造(Problem / Context / Alternatives / 検証ゴール)を持つ issue ほど「完成して見える」。本 spec の仕事は、その**整った構造が隠している未決定**を体系的に尋問し、着手前に差し出すこと。

- **やる**: issue を読み・grounding し・8 観点を当て・優先度付きレビュー本文を生成する。
- **やらない(dispatch wrapper / orchestrator の責務)**: コメント投稿 / マーカー埋込 / ラベル付与 / issue 編集 / close / 台帳書込。本 spec は**レビュー本文を組み立てて返すだけ**(副作用ゼロ)。投稿・blocker_count 計数・escalate 補完・marker 埋込は `roles/issue-reviewer-dispatch.md` が担う。
- **足さない**: 構造が飽和した issue に*さらに構造*を足すのは仕事ではない。突くのは中身と判断だけ。

## ワークフロー

1. **取得**: `gh issue view <n> --repo <owner/repo> --json number,title,body,labels,comments` で取る。既存コメント(過去レビュー含む)も読み、**重複指摘を避ける**。
2. **grounding**: issue 本文が参照する docs / コード / 関連 issue を読む。CWD が対象リポジトリのクローンなら該当ファイルを Read、無ければ issue 本文 + `gh issue view <related>` / `gh api` で取れる範囲。**空中戦で当てない**。grounding できなかった範囲は前提として明記する。
3. **8 観点を当てる**: 下のルーブリック(問い / なぜ効く / ミスの兆候 / 汎用症状例)を各観点で当て、「未決定」を探す。
4. **校正**: severity を正確に。誇張しない。🔴 は本当に効くものだけに絞る(9 個出すより効く 3 個)。良い箇所は良いと明記する。
5. **生成**: 下の出力フォーマットでレビュー本文を作る。**投稿しない**(投稿は dispatch wrapper)。
6. **返す**: dispatch wrapper へレビュー本文(`review_markdown` 相当)と `has_blocker`(🔴 が 1 つでもあれば true)を返す。以降の blocker_count / escalate / marker / 投稿の補完と `contracts/issue-reviewer-return.schema.json` の形での返却は dispatch wrapper が行う。

## 8 観点 / 3 ファミリー ルーブリック

各観点は「問い / なぜ効く / ミスの兆候 / 汎用症状例」で構成する。**汎用症状例は repo 非依存の一般化した症状**であり(特定リポジトリの実 issue 番号ではない)、対象 issue のドメインに合わせて読み替える — 例そのものを当てるのではなく、同じ**形の未決定**が対象 issue に無いかを探すための当て木として使う。

### ファミリー1 ── 「完了」は本物か

#### L1. DoD ↔ Problem 一致
- **問い**: 検証ゴールを全部満たしたとき、Problem が約束した*価値*は証明されるか。それとも*手段*の確認止まりか。代理指標にすり替わっていないか。
- **なぜ効く**: 構造が整った issue ほど「検証ゴール」欄が埋まっているので、満たせば完了に*見える*。だが手段(動く / 署名が通る / 緑になる)の確認と、価値(ユーザーの課題が消える)の証明は別物。
- **ミスの兆候**: 検証ゴールが動作確認・ビルド成否・lint 緑など*手段*ばかりで、Problem の中心語(「素早く」「自動で」「連携する」)に対応する項目が無い。
- **汎用症状例**: Problem の核が「外部イベントとの連携」なのに、DoD が「起動する / 署名検証が通る」のような手段の確認だけで、連携そのものの成否を測る項目が無く、核心が別マイルストーンへ先送りされている(= その単位は『成功』しても Problem が約束した価値は未証明のまま)。

#### L2. テストを書けるまで仕様が降りているか
- **問い**: 限界値・境界・分岐の*期待挙動*が、そのままテストに落とせるほど具体に書かれているか。
- **なぜ効く**: テスト駆動を掲げる単位ほど、曖昧な限界値はテストを書けず、方針そのものと矛盾する。曖昧仕様は実装者裁量に落ち、後で「そういう意図ではなかった」になる。
- **ミスの兆候**: 「上限」「安全弁」「適切に」「いい感じに」等が、具体的な*挙動*に翻訳されていない。同値・順序・タイブレークが未定義。
- **汎用症状例**: 「上限 N 件」と書くが超過時の挙動が未定義。走査順で先着 N 件を採ると非決定的になり、「決定論的にテストできる」という採用根拠と自己矛盾する(境界の期待挙動が未定義のまま「安全弁がある」と書かれている)。

### ファミリー2 ── 「決定」は本物か

#### L3. 決定の質 + 検証済みの逃げ道
- **問い**: Alternatives は本物の対立案か(藁人形を並べて本命を正当化していないか)。採用案のコストは有界か。ダメだったときの逃げ道は*実在を確認済み*か。
- **なぜ効く**: 「選択肢を並べた」こと自体は決定の質を保証しない。特に**検証していないフォールバックは、フォールバックではない** — 撤退先が幻なら、その決定は逃げ道の無い賭けと同じ。
- **ミスの兆候**: 不採用案の demerit が誇張・空疎。採用案の demerit が「罠を踏む」程度で*いつ撤退するか*の基準が無い(コスト青天井)。「〜なら別案」と書いてあるが別案の実在を確認していない。
- **汎用症状例**: フォールバック「機能 X が使えなければ代替 Y を使う」と書くが、対象環境に Y が存在するかを未確認で、実装後に幻だったと判明する。または採用案の demerit に「いつ諦めるか」の bail 基準が無くコストが有界化されていない。

#### L4. 事実の顔をした仮定
- **問い**: 断定文の中に、実測していない仮定が混ざっていないか。許容 / 失敗の閾値はあるか。
- **ミスの兆候**: 「体感に問題が出にくい」「十分速い」「めったに起きない」等を、根拠・閾値なしに既定事実として扱っている。
- **汎用症状例**: 「毎回フル走査でも体感に問題が出にくい」は未実測の性能仮定。許容レイテンシの閾値も、それを超えたときの代替も書かれていない(断定の顔をしているが検証されていない)。

#### L5. 強制の比例と閉ループ
- **問い**: その制約は L1(指示)〜 L4(不変条件テスト)のどの層にあるか。リスクに比例しているか(過剰武装 / 過少防御)。そして*実際に強制*されているか — ループは閉じているか。
- **なぜ効く**: ゲートや規約は*存在する*だけでは効かない。強制されて初めて feedback ループが閉じる。逆に低リスクに最重量の層を盛ると保守倒れする。
- **ミスの兆候**: 「ゲートはあるが手で叩く前提」「規約はあるが検知が無い」。または重要でない対象に最重量の層を当てている。
- **汎用症状例**: `make check / test / build` は用意されているが CI の必須チェックが空で未強制(ループ未閉 = 規約が存在するだけ)。逆に、低リスクの対象に初日からフル装備の検査層を積んで保守過多になっている(比例していない)。

### ファミリー3 ── 文脈で壊れないか

#### L6. 失敗・空・変化状態 + 列挙の網羅性
- **問い**: happy path 以外(失敗・空・権限拒否・外界が変わった)を扱っているか。閉じた集合(脅威 / 除外 / 入力種別)を主張したなら、それは本当に閉じているか。
- **ミスの兆候**: 正常系だけ厚い。永続化した状態が「次回も同じ世界」を暗黙に仮定。「対象は N 種類」「経路は M 個」の列挙に抜けがある。
- **汎用症状例**: 状態(パス等)を永続化するが、その対象が削除 / 移動された後の再起動(stale な参照)の挙動が未定義。または「漏洩経路は 4 つ」のような閉集合の主張に、実は数え漏れた経路がある。

#### L7. 環境・移植性
- **問い**: 開発機の外で成立するか — CI runner / 別 OS バージョン / 別ユーザー / sandbox 有無。依存するバージョンはピンされているか。
- **なぜ効く**: 「自分の環境では動く」は証拠にならない。環境差は*コード起因でない*失敗を生み、原因特定が遅れる。
- **ミスの兆候**: ローカル前提の挙動(パス導出・権限・大文字小文字)を不変と仮定。「latest」依存でバージョンを固定していない。
- **汎用症状例**: ローカルでは通る手順が、CI runner の toolchain バージョン差 / 別 OS のファイルシステム(大文字小文字の扱い)/ フル環境と最小環境のパス導出差で割れる。「緑になる」と書くがバージョン未ピンで後日 runner drift で崩れる。

#### L8. リスクの順序付け・依存の健全性・継承の未検証
- **問い**: 最大の不確実性が前倒しされているか。前提(依存 issue / PR)は実在し、順序は確定しているか。前マイルストーン・AI・ツール既定から*継承した挙動*で、未検証のものはないか。
- **なぜ効く**: 怖い所を後回しにすると手戻りがマイルストーン数個分になる。継承した未検証物は「誰も検証していない前提」として静かに残る。
- **ミスの兆候**: 一番怖い未知が後ろの単位に置かれている。依存の順序が「承諾任せ」で未確定。「前の人 / AI / ツールがやった」挙動を検証なしに土台にしている。
- **汎用症状例**: 依存する前工程(別 issue / 別 PR / 別マイルストーン)の完了順が「承諾任せ」で未確定のまま後続を設計している。または前工程で自動ツール(コード生成 AI 等)が入れた挙動を未検証のまま土台にして、後工程の CI に乗せている。

## 出力フォーマット

冒頭 1 行:「構造は良い。以下は構造が隠した未決定を 8 観点で洗ったもの。採否は作者判断」。指摘は 3 段、各々〔**観点ラベル(L番号)+ ギャップ + 〈決めるべきこと〉(actionable)**〕で書く:

- 🔴 **着手前ゲート** — テストが書けない / Problem 直結 / 致命的な未決定。実装前に潰す。
- 🟡 **仕様を 1 行ずつ確定** — テスト設計時に決めれば足りる境界・分岐。
- 🟢 **スコープ判断** — 今回扱うか否か。扱わないなら「この issue では扱わない」と明記(暗黙の未対応にしない)。

末尾 1 行:どの観点(ファミリー / L番号)で当てたか / 修正は当てていない(採否は作者)。良い箇所は良いと明記する。
複数回レビューする issue では、既出指摘を繰り返さず**増分だけ**出す(過去コメントを参照)。

## 原則

1. 構造を足さない、substance を突く。
2. 暗黙知を毎回標準で当てる(気づいたときだけ、にしない)。
3. 誇張しない。過大主張は過大と評価し直す。
4. doer ≠ judge。レビューは修正を当てず、採否は作者が判断する。
5. 意図的除外を明示。書かれない未対応は後で「気づかなかった」になる。
6. 差し出すが、決めない。観点は決定を*促す*ためにあり、人間の判断を代替しない。

## kit 版 rubric と個人 skill の drift(best-effort・正直な明記・issue #93)

本 spec(kit 同梱の既定モード)と、opt-in モードが委譲する個人 skill `reviewing-github-issues` は、**同じ 8 観点 rubric の 2 系統**になる。個人 skill は repo 外(`~/.claude/skills`)にあり **smoke / CI から読めない**ため、両系統の rubric drift を機械検知することは**原理的に不可能**(PR 側 `collectors/angles/*` の CANONICAL_CORE のような逐語 drift 検査は掛けられない)。したがって同期は **best-effort の手動同期**とする — どちらかの観点(問い・兆候・例)を改善したら、もう一方も手で追随させる。これは issue フェーズに evidence gate が構造的に無いこと・`reports[]` が best-effort であることと同じ流儀の**受容コスト**であり、暗黙の乖離リスクのまま残さないためここに明記する。

## I/O 契約

- **入力**: `repo`(owner/name) + issue 番号。
- **出力(本 spec)**: レビュー本文 + `has_blocker`(🔴 が 1 件でもあれば true)。投稿・マーカー・ラベルは行わない。
- **dispatch wrapper への引き渡し**: `roles/issue-reviewer-dispatch.md` が本 spec の出力を受け、blocker_count(🔴 件数)・escalate(`scripts/evaluate-stop-condition.py`)・marker 埋込・投稿を補完し、`contracts/issue-reviewer-return.schema.json` の形(`{has_blocker, blocker_count, escalate, review_markdown}`、任意で `escalate_to_human`)で orchestrator へ返す。台帳・ラベルへの書込は単一書込主体の orchestrator の専権。

台帳ファイル(`.harness/plan-progress.json`)には一切触れるな(Read も編集もするな)。
